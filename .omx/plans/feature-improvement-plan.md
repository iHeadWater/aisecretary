# 功能改进计划

## 问题 1：跨 Context 事务感知缺失

与 aisecretary 在不同 context 中创建的事务会记录到同一个数据库，但在一个 context 中，aisecretary 对其他 context 创建的事务没有印象；在一个 context 中对话过长会导致前文被压缩，aisecretary 对该 context 中的事务也没有印象。

## 问题 2：隐私泄露风险

在群聊中，成员可能通过与群聊中的 aisecretary 对话，将此 aisecretary 创建者的隐私信息调取出来（如个人事务、联系方式、财务信息等）。

## 问题 3：本地运行安全风险

目前 Hermes 运行在本机上，直接暴露在网络环境中，存在被黑客入侵的风险。需要将 Hermes 部署 Docker 化以实现隔离。

## 问题 4：事务防篡改

创建的事务不能删除，事务的状态也不能修改为"已取消"。需要确保事务记录不可篡改和不可删除，以确保记录的真实性和完整性。

## 问题 5：事务展示方式

aisecretary 不能把事务以 markdown 表格的形式输出给用户，这会导致信息密度过高，用户难以阅读。应以简洁的列表形式展示事务。

## 问题 6：对话历史存储

考虑到 Hermes 拥有记忆能力，面对大量的对话内容，是否需要数据库备份存储对话内容，以实现跨 Context 的完整对话历史追溯。

## 问题 7：事务关联

计划将创建的事务之间建立起关联，实现去原子化。将相关事务关联起来可以更好地组织和管理事务。

## 问题 8：数据库管理方式

管理数据库是采用 SQLite 的 MCP 服务还是代码 + Skill 的方式，需要权衡利弊。
- MCP 服务：统一管理，但增加了外部依赖
- 代码 + Skill：更灵活，但需要自行实现管理逻辑

## 问题 9：多 aisecretary 共存

如何实现在同一群聊中接入来自不同用户的 aisecretary（如 OpenAI、Hermes），同时需要考虑处理多个 aisecretary 之间的协作和隔离问题。
